はじめかたGetting started

AI と作ったウェブツールに、住む場所を。 Give your AI-built web tools a home.

kotoji は、あなた（とあなたの AI）が作る小さなウェブツールのための、 MCP ネイティブなセルフホスト基盤です。HTML/CSS/JS のフォルダを置けば きれいな URL が手に入り、ブラウザ上で編集でき、git がすべての変更を 静かにバージョン管理します。 kotoji is an MCP-native, self-hosted home for the small web tools you (and your AI) build. Drop a folder of HTML/CSS/JS, get a clean URL, edit it in the browser, and let git quietly version every change.

名前は兼六園の琴柱灯籠（ことじとうろう）に由来します。 琴柱（琴柱）は琴の弦を支える柱です。kotoji もまた、 その上に置かれたツールを支え、声を与える存在です。 The name comes from the kotoji-tōrō lantern of Kenrokuen garden. Kotoji (琴柱) is the bridge that supports the strings of a koto — just as kotoji supports and gives voice to the tools you place on it.

5 ステップで始める → Start in 5 steps → AI を接続する Connect your AI

1サイトを作成Create a site

ダッシュボードで New を選び、handle （ツールが住むサブドメイン）を決めます。handle は 小文字・数字・ハイフンが使えます（例: expense-calc）。予約語リストにより mcp や api などの名前は使えません。 From the dashboard, choose New and pick a handle — the subdomain your tool will live on. Handles are lowercase letters, numbers and hyphens (e.g. expense-calc). A reserved-word list keeps names like mcp and api off-limits.

2コンテンツを追加Add content

ファイルをサイトに入れる方法は対等に 3 通りあります。いずれも同じ git の境界を通るので、すべての変更がコミットになります。 There are three peer ways to put files into a site — and they all funnel through the same git boundary, so every change is a commit.

.zip をドラッグDrag a .zip

HTML/CSS/JS のアーカイブをダッシュボードにドロップします。kotoji がワーキングツリーへ展開します。 Drop an archive of HTML/CSS/JS onto the dashboard. kotoji unpacks it into the working tree.

ブラウザで編集Edit in the browser

Monaco エディタを開いて変更し、Save を押します。各 Save が差分付きのコミットになります。 Open the Monaco editor, make a change, and hit Save — each Save is a commit with a diff.

AI に書かせるLet your AI write

MCP クライアントを接続（ステップ 6 参照）すれば、AI が代わりにファイルを読み書きします。同じ git、同じ履歴です。 Connect an MCP client (see step 6) and it reads & writes files for you. Same git, same history.

Monaco エディタはデスクトップ/タブレット向けです。スマホでは 読みやすいビューアと「大きい画面で開く / AI に頼む」という代替手段が 表示されます。公開・履歴・バージョンの操作はスマホでも完全に使えます。 The Monaco editor is a desktop/tablet experience; on phones you get a read-friendly viewer and an “open on a bigger screen / ask AI” fallback — publish, history and versions stay fully usable on a phone.

3下書きと公開Draft vs. Published

作業は常に draft（下書き）ブランチで行います — 安全な 作業場所です。そこで保存したものは、まだ公開されません。 You always work on the draft branch — your safe workspace. Nothing you save there is live yet.

  • 下書き（draft） — 既定の作業ブランチ。自由に編集・アップロード・試行できます。 Draft — the default working branch. Edit, upload and experiment freely.
  • 公開（Publish） — draft を published へ昇格させます。これが your-handle.example.com で訪問者に見えるものです。 Publish — promotes draft to published, which is what visitors see at your-handle.example.com.
  • 履歴とロールバック — すべての保存と公開がコミットなので、差分を確認し、いつでも過去のバージョンへ巻き戻せます。 History & rollback — every save and publish is a commit, so you can review the diff and roll back to any earlier version at any time.

ここでは git が唯一の真実です。バージョン管理は kotoji の保存方法の副産物であり、別途管理すべき機能ではありません。 git is the single source of truth here — versioning is a side effect of how kotoji saves, not a separate feature you have to manage.

4バージョンとプレビューVersions & previews

draft を触らずに大きめの変更を試したいときは、別の バージョン（ブランチ）を作ります。各ブランチには 専用のプレビュー URL が付くので、昇格前にライブで 確認できます。 Need to try a bigger change without touching draft? Create another version (a branch). Each branch gets its own per-branch preview URL so you can see it live before promoting: 

published   →  your-handle.example.com
draft       →  your-handle--draft.example.com
feature-x   →  your-handle--feature-x.example.com

プレビューのホスト名は handle--branch 形式です。 非公開サイトのプレビューは短命なプレビュー許可で保護されるので、 公開前の作業中の内容が露出することはありません。 Preview hosts use the handle--branch form. Previews of non-public sites are gated by a short-lived preview grant, so work-in-progress isn't exposed before you publish it.

5公開範囲Visibility

各サイトには、配信コンテンツに誰がアクセスできるかを制御する 公開範囲のティアがあります。新しいサイトは既定で Private（非公開）です — 広い公開はオプトインです。 Each site has a visibility tier that controls who can reach its served content. New sites are Private by default — you opt in to wider access.

Public — URL を知る誰でもanyone with the URL Members — サイトにサインインしたメンバーsigned-in members of the site Private — オーナー / メンバーのみowner / members only

公開範囲はサイトごとに設定され、データプレーン（配信されるページ）を 統制します。コントロールプレーンへのアクセスは、ティアに関係なく常に メンバーシップで保護されます。 Visibility is per-site and governs the data plane (the served pages); control-plane access is always membership-gated regardless of tier.

6MCP で AI を接続Connect your AI over MCP

kotoji は MCP ネイティブです。あなたの AI が、サイトを直接 一覧・読み取り・書き込み・公開できます。設定は一度きりです。 kotoji is MCP-native: your AI can list, read, write and publish your sites directly. Set it up once.

  1. Settings を開き、ユーザー単位の MCP/API トークンを発行します（一度だけコピー — あなたがメンバーである全プロジェクトにまたがります）。 Open Settings and issue a per-user MCP/API token (copy it once — it spans every project you're a member of).
  2. MCP クライアント（例: Claude）を https://example.com/mcp に向け、そのトークンを Bearer 認証情報として渡します。 Point your MCP client (e.g. Claude) at https://example.com/mcp with that token as a Bearer credential.
  3. すべてのツールは サイトの handle セレクタを取るので、1 つのトークンで全サイトを操作できます。 Every tool takes a site handle selector, so one token works across all your sites.

クライアント設定の例 — origin を置き換え、実際のトークンを貼り付けてください: Example client config — replace the origin and paste your real token: 

{
  "mcpServers": {
    "kotoji": {
      "url": "https://example.com/mcp",
      "headers": {
        "Authorization": "Bearer kotoji_pat_REPLACE_WITH_YOUR_TOKEN"
      }
    }
  }
}

AI が使えるツール（注記がない限り、それぞれ site handle を取ります）: The tools your AI gets (each takes a site handle, except where noted):

read

調べるExplore

list_sites (handle 不要no handle)、list_files、read_file、get_diff、get_log。

write

作るBuild

write_file、save、create_branch、rollback — すべて draft ブランチ上で。all on the draft branch.

publish

出荷するShip

publish は draft をライブへ昇格します。create_site は新しいサイトを発行します（トークンに権限がある場合）。 publish promotes draft to live. create_site mints a new site (when your token may).

あるサイトにおけるトークンの権限は token.scopes ∩ そのサイトでのあなたのロール で、 呼び出しのたびに再チェックされます。メンバーシップを外せば、トークンの 権限は即座に制限されます。 The token's power on any site is token.scopes ∩ your-role-on-that-site, re-checked on every call — so removing a membership instantly limits the token.

任意のオプションOptional extras

GitHub ミラーGitHub mirror

管理者は Settings で GitHub ミラーを有効化できます。公開のたびに、外部バックアップとしてミラープッシュされます。PAT は暗号化して保存されます。 An admin can enable a GitHub mirror in Settings; every publish mirror-pushes as an off-box backup. The PAT is stored encrypted at rest.

オプトインの Traefik オーバーレイを追加すると、ベースドメイン向けのワイルドカード TLS 証明書を自動発行する、単体で完結する構成になります。 Add the opt-in Traefik overlay for a self-contained box that auto-issues a wildcard TLS cert for your base domain.
