Getting started

Give your AI-built web tools a home.

kotoji is an MCP-native, self-hosted home for the small web tools you (and your AI) build. Drop a folder of HTML/CSS/JS, get a clean URL, edit it in the browser, and let git quietly version every change.

The name comes from the kotoji-tōrō lantern of Kenrokuen garden. Kotoji (琴柱) is the bridge that supports the strings of a koto — just as kotoji supports and gives voice to the tools you place on it.

Start in 5 steps → Connect your AI

1Create a site

From the dashboard, choose New and pick a handle — the subdomain your tool will live on. Handles are lowercase letters, numbers and hyphens (e.g. expense-calc). A reserved-word list keeps names like mcp and api off-limits.

Empty

Empty

Start blank, then add files in the browser or over MCP. Best when your AI will build it for you.

From a zip

From a zip

Upload a .zip of static files and serve it instantly — exactly how this guide gets published.

Template

Template

Scaffold from a starter template to get a working page without writing the boilerplate yourself.

Your tool will be reachable at your-handle.example.com once published (swap example.com for your instance's base domain).

2Add content

There are three peer ways to put files into a site — and they all funnel through the same git boundary, so every change is a commit.

Drag a .zip

Drop an archive of HTML/CSS/JS onto the dashboard. kotoji unpacks it into the working tree.

Edit in the browser

Open the Monaco editor, make a change, and hit Save — each Save is a commit with a diff.

Let your AI write

Connect an MCP client (see step 6) and it reads & writes files for you. Same git, same history.

The Monaco editor is a desktop/tablet experience; on phones you get a read-friendly viewer and an “open on a bigger screen / ask AI” fallback — publish, history and versions stay fully usable on a phone.

3Draft vs. Published

You always work on the draft branch — your safe workspace. Nothing you save there is live yet.

  • Draft — the default working branch. Edit, upload and experiment freely.
  • Publish — promotes draft to published, which is what visitors see at your-handle.example.com.
  • History & rollback — every save and publish is a commit, so you can review the diff and roll back to any earlier version at any time.

git is the single source of truth here — versioning is a side effect of how kotoji saves, not a separate feature you have to manage.

4Versions & previews

Need to try a bigger change without touching draft? Create another version (a branch). Each branch gets its own per-branch preview URL so you can see it live before promoting: 

published   →  your-handle.example.com
draft       →  your-handle--draft.example.com
feature-x   →  your-handle--feature-x.example.com

Preview hosts use the handle--branch form. Previews of non-public sites are gated by a short-lived preview grant, so work-in-progress isn't exposed before you publish it.

5Visibility

Each site has a visibility tier that controls who can reach its served content. New sites are Private by default — you opt in to wider access.

Public — anyone with the URL Members — signed-in members of the site Private — owner / members only

Visibility is per-site and governs the data plane (the served pages); control-plane access is always membership-gated regardless of tier.

6Connect your AI over MCP

kotoji is MCP-native: your AI can list, read, write and publish your sites directly. Set it up once.

  1. Open Settings and issue a per-user MCP/API token (copy it once — it spans every project you're a member of).
  2. Point your MCP client (e.g. Claude) at https://example.com/mcp with that token as a Bearer credential.
  3. Every tool takes a site handle selector, so one token works across all your sites.

Example client config — replace the origin and paste your real token:

{
  "mcpServers": {
    "kotoji": {
      "url": "https://example.com/mcp",
      "headers": {
        "Authorization": "Bearer kotoji_pat_REPLACE_WITH_YOUR_TOKEN"
      }
    }
  }
}

The tools your AI gets (each takes a site handle, except where noted):

read

Explore

list_sites (no handle), list_files, read_file, get_diff, get_log.

write

Build

write_file, save, create_branch, rollback — all on the draft branch.

publish

Ship

publish promotes draft to live. create_site mints a new site (when your token may).

The token's power on any site is token.scopes ∩ your-role-on-that-site, re-checked on every call — so removing a membership instantly limits the token.

Optional extras

GitHub mirror

An admin can enable a GitHub mirror in Settings; every publish mirror-pushes as an off-box backup. The PAT is stored encrypted at rest.

Automatic HTTPS

Add the opt-in Traefik overlay for a self-contained box that auto-issues a wildcard TLS cert for your base domain.
